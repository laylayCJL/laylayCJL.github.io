    一、数据和数据模型

    1、数据与信息

    （1）基本概念

    1）数据：描述事物的符号记录

    • 数据有一定的格式，这些格式的规定是数据的语法，而数据的含义是数据的语义
    • 数据是信息存在的一种形式，只有通过解释或处理才能成为有用的信息

    2）信息：从数据中获得的有意义的内容

    eg：数据：张三，1912，男，1981，计算机系，应用软件

    —(解释)—>信息：张三是1912班的男生，1981年出生，计算机系应用软件专业

    （2）数据的静态特征与动态特征

    1）静态特征

    graph LR
H(静态特征)-->A(数据的基本结构)
H(静态特征)-->D(数据间的联系)
H(静态特征)-->F(数据取值范围约束)
A(数据的基本结构)-->B(eg.学生学号,姓名...)
D(数据间的联系)-->E(eg.学生选课中的学号与学生基本信息中的学号)
F(数据取值范围约束)-->G(eg.年龄>0,成绩<100,...)< pre>


    2）动态特征
    

    graph LR
H(动态特征)-->A(数据可以进行的操作)
H(动态特征)-->D(操作规则)
A(数据操作)-->B(增删改查)
D(操作规则)
    


    2、数据模型
    数据模型(data model)是对现实世界数据特征的抽象，是数据库系统中用于提供信息表示和操作手段的形式架构。
    

    （1）数据模型的三要素
    graph LR
H(数据模型的三要素)-->A(数据结构)
H(数据模型的三要素)-->D(数据操作)
H(数据模型的三要素)-->F(数据的完整性约束条件)
    




    1）数据结构
    

      

    • 数据结构是所描述的对象类型的集合。现在常用的是关系数据模型
      • 

    • 数据结构是对系统静态特性的描述
      • 

    

    2）数据操作
    

      

    • 数据操作是指对数据库表中记录的值允许执行的操作集合，数据库对数据的操作主要有增、删、改、查询4种操作
      • 

    • 数据操作是对系统动态特性的描述
      • 

    

    3）数据的完整性约束条件
    

      

    • 数据的完整性约束条件是一组完整性规则。用以保证数据的正确、有效、相容。
      • 

    

    （2）数据模型分类
    graph LR
H(数据模型)-->A(概念层数据模型: 语义视角抽象)
H(数据模型)-->D(组织层数据模型: 组织层面描述)
    


    

      

    • 概念层数据模型（概念模型）：从数据的语义视角来抽取模型，是按用户的观点来对数据和信息进行建模
        

      • 主要用在数据库的设计阶段
        • 

      • 与具体的数据库管理系统无关
        • 

      • 与具体的实现方式无关
        • 

      • 常用概念模型：实体-联系模型、语义对象模型
        • 

      

      • 

    • 组织层数据模型（组织模型）：从数据的组织层次来描述数据。
        

      • 是从计算机系统的观点对数据进行建模,与所使用的数据库管理系统有关
        • 

      • 常用组织模型：层次模型、网状模型、关系模型、对象-关系模型
        • 

      

      • 

    

    二、概念层数据模型
    是面向用户、面向现实世界的数据模型，与具体的DBMS无关
    

    1、基本概念
    （1）概念层数据模型完成的任务：
    

      

    • 抽象现实系统中有应用价值的元素及其关联关系
      • 

    • 反映现实系统中有应用价值的信息结构
      • 

    • 不依赖于数据的组织层数据模型
      • 

    

    （2）概念层数据模型存在的意义：
    

      

    • 是现实世界到信息世界的第一层抽象
      • 

    • 是数据库设计人员进行数据库设计的工具
      • 

    • 是数据库设计人员和业务领域的用户之间进行交流的工具
      • 

    

    （3）对概念层数据模型的要求：
    

      

    • 具有较强的语义表达能力
      • 

    • 能够方便、直接地表达应用中的各种语义知识
      • 

    • 简单、清晰，易于被用户理解
      • 

    

    2、实体-联系模型（Entity-Relationship Model）
    （1）实体-联系模型基本简介
    1）基本观点
    

    世界是由一组称作实体的基本对象和这些对象之间的联系构成
    

    2）E-R模型的三要素
    

    实体型、属性和联系
    

    3）E-R图
    

    矩形框——实体型，菱形框——联系类型，椭圆型框——实体与联系类型的属性
    

    直线——实体型与联系之间用直线相连，线旁边标注联系的种类
    

    eg.
    

    

    （2）实体（名词）
    具有公共性质的可相互区分的现实世界对象的集合，eg.学生、课程、职工
    

      

    • 实体在E-R图的表示方法：用矩形框表示具体的实体，把实体名写在框内
      • 

    

    （3）属性（名词的属性）
    描述实体或者联系的性质或特征的数据项，eg.学号、姓名、性别、年龄
    

      

    • 标识属性：能够唯一标识实体的一个属性或最小的一组属性（称为属性集或属性组）称为实体的标识属性，又称为实体的码。
      例如，“学号”就是学生实体的码
      • 

    • 属性在E-R图的表示方法：用圆角矩形或椭圆框表示，框内写上属性名，并用连线连到相应实体
      • 

    

    （4）联系（动词：名词间的联系）
    联系是数据之间的关联集合，是客观存在的应用语义链
    

      

      • 

      • 实体内部的联系：一个实体内属性之间的联系。职工中的职工号和此职工的部门经理号
        • 

      • 实体之间的联系：不同实体之间的联系。课程和学生实体之间存在选课联系
        • 

      

      • 

    • 联系在E-R图的表示方法：实体之间的联系用菱形框表示，框内写上联系名，并用连线与有关的实体相连
      

      • 

    • 联系的种类：
      

        

      • 一对一联系(1:1)
        

        如果实体A中的每个实例在实体B中至多有一个（也可以没有）实例与之关联，反之亦然，则称实体A与实体B具有一对一联系，记作：1:1 
        

        eg. 
        

            graph TD
a[班长]-->|1|b{管理}
b{管理}-->|1|c[班级]
        




        • 

      

      • 

    

      

    • 一对多联系(1:n)
      

      如果实体A与实体B之间存在联系，并且对于实体A中的一个实例，实体B中有多个实例与之对应；而对实体B中的任意一个实例，在实体A中都只有一个实例与之对应，则称实体A到实体B的联系是一对多的，记为1 : n 
      

      eg.
      

          graph TD
a[班长]-->|1|b{管理}
b{管理}-->|n|c[班级成员]
      

      • 

    • 多对多联系(m:n)
      

      如果实体A与实体B之间存在联系，并且对于实体A中的一个实例，实体B中有多个实例与之对应；而对实体B中的一个实例，在实体A中也有多个实例与之对应，则称实体A到实体B的联系是多对多的，记为m : n 
      

      eg.
      

          graph TD
a[班干部]-->|m|b{管理}
b{管理}-->|n|c[班级成员]
      

      • 

    • 区分：关联多个实体的多对多联系 与 关联两个实体的多对多联系
      

      

      

      • 

    

    三、组织层数据模型
      

    • 组织层数据模型是从数据的组织形式的角度来描述信息
      • 

    • 在数据库技术的发展过程中用到的组织层数据模型主要有：
        

      • 层次模型（Hierarchical Model）
        • 

      • 网状模型（Network Model）
        • 

      • 关系模型（Relational Model）
        • 

      • 面向对象模型（Object Oriented Model）
        • 

      

      • 

    

    1、层次数据模型
    1）简介
    

      

    • 用树形结构表示实体和实体之间的联系。
      现实世界中许多实体之间的联系本身就呈现出一种自然的层次关系，如：行政机构、家族关系
      

      • 

    • 构成层次模型的树由结点和连线组成
      

        

      • 结点表示实体，结点中的项表示实体的属性
        • 

      • 连线表示相连的两个实体间的联系，这种联系是一对多的。
        • 

      

      • 

      • 

      • 通常把表示“一”的实体放在上方，称为父结点
        • 

      • 把表示“多”的实体放在下方，称为子结点
        • 

      • 将不包含任何子结点的结点称为叶结点
        • 

      

      • 

    

    

    2）层次模型的要求
    

      

    • 有且仅有一个结点无父结点，这个结点即为树的根
      • 

    • 其他结点有且仅有一个父结点
      • 

    

    3）层次模型基本特点
    

      

    • 任何一个给定的记录值只有从层次模型的根部开始按路径查看时，才能明确其含义，任何子结点都不能脱离父结点而存在
      • 

    

    4）层次模型局限性
    

      

    • 只能表示一对多的联系，不能直接表示多对多联系
      • 

    • 如果把多对多联系转换为一对多联系，会出现一个子结点有多个父结点的情况
      •

    • 如果去掉层次模型中的两点限制，即：①允许一个以上的结点无父结点②每个结点可以有多个父结点便构成了网状模型

    2）网状模型的局限
    

      

    • 网状数据模型可以直接表示多对多联系。但实现起来太复杂。因此一些支持网状模型的数据库管理系统，对多对多联系还是进行了限制
      

      例如，网状模型的典型代表CODASYL就只支持一对多联系
      

      •

    2）关系模型好处
    

      

    • 关系数据模型易于设计、实现、维护和使用

    3、NoSQL
    （1）概述
    

     NoSQL是非关系型数据存储的广义定义。它打破了长久以来关系型数据库与事务ACID理论大一统的局面。NoSQL数据存储不需要固定的表结构，通常也不存在连接操作。在大数据存取上具备关系型数据库无法比拟的性能优势
    

    （2）NoSQL相关理论
    

      

    • CAP理论
      

        

      • 是NoSQL数据库的基石
        

        • 

      • CAP理论指的是在一个分布式系统中，Consistency（一致性）、 Availability（可用性）、Partition tolerance（分区容忍性），三者不可兼得
        

        一个分布式系统无法同时满足一致性、可用性、分区容忍性三个特点，最多只能实现其中两点。而由于当前的网络硬件肯定会出现延迟丢包等问题，所以分区容忍性是我们必须需要实现的。所以我们只能在一致性和可用性之间进行权衡，没有NoSQL系统能同时保证这三点
        

        • 

      

      • 

    • BASE理论
      

      Basically Available（基本可用）、Soft state（软状态）和Eventually consistent（最终一致性）三个短语的简写，BASE是对CAP中一致性和可用性权衡的结果，其核心思想是即使无法做到强一致性，但每个应用都可以根据自身的业务特点，采用适当的方式使系统达到最终一致性
      

      • 

    

    五、数据库结构
    ==数据库的体系结构：三级模式和两级映像==
    

    1、模式的基本概念
    1）模式是数据库中全体数据的逻辑结构和特征的描述，它仅仅涉及“型”的描述，不涉及具体的值
    

    2）关系模式是关系的“型”或元组的结构共性的描述
    

    实际上对应的是关系表的表头
    

    

      

    • 实例与模式
        

      • 模式的一个具体值——>实例（instance），所以一个模式可以对应多个实例
        • 

      

      • 

    

    graph TD
A(模式)-->B(稳定的)
A-->|反映|C(数据的结构与联系)
E(实例)-->F(变动的)
E(实例)-->|反映|D(数据库某一时刻的状态)
    


    2、三级模式结构
      

    • 分三级模式的目的：
      

        

      • 为了提高数据的物理独立性和逻辑独立性，使数据库的用户观点（即用户看到的数据库）与数据库的物理方面（即实际存储的数据库）区分开来
        • 

      • 数据库系统的三级模式是对数据的三个抽象级别，它把数据的具体组织留给DBMS管理，使用户能逻辑地、抽象地处理数据，而不必关心数据在计算机中的具体表示方式与存储方式
        • 

      

      • 

    • 三级模式示意图
      

      • 

    

     
    

    graph LR
A(模式)-->B(外模式 = 子模式 = 用户模式)
A(模式)-->C(模式 = 逻辑模式 = 概念模式)
A(模式)-->D(内模式 = 存储模式 = 物理模式)

    • 外模式是模式的子集解释：
      

      举例如下：

    • 子模式可以重构解释：
      

      学生（学号，姓名，性别，年龄，所在系）
      课程（课程号，课程名，学分）
      选课（学号，课程号，成绩）
      

      三者可以重构为：学生（姓名，课程名，成绩）
      

      • 

    • 外模式的安全性更强
      

      因为外模式是模式的子集，所以有一些信息可以隐藏起来，不必全部展出，比如学生的身份证号属于个人隐私，可以不把身份证号这一项放在外模式中，从而实现隐藏
      

      • 

    

    2）模式
    

    3）外模式
    

      

    • 内模式是对整个数据库的底层表示。描述了数据的存储结构，但不涉及物理记录的形式。比如：
        

      • 数据的组织与存储方式
        • 

      • 索引按什么方式组织
        • 

      • 是否加密
        • 

      

      • 

    

    4）模式小结
      

    • 模式是数据库的中心与关键，它独立于数据库系统的其他层。设计数据库系统时也是首先设计数据库系统的逻辑模式。
      • 

    • 内模式依赖于数据库的全局逻辑结构，但独立于数据库的用户视图，也独立于具体的存储设备。
      • 

    • 外模式面向具体的应用程序，它定义在逻辑模式之上，但独立于存储模式和存储设备。
      • 

    

    3、模式映像与数据独立性
      

    • 两级映像的目的：为了能够在内部实现这三个抽象层次的联系和转换，数据库管理系统在三级模式之间提供了两层映象：
        

      • 外模式/模式映象
        • 

      • 模式/内模式映象
        • 

      

      • 

    

    1）外模式/模式映象
      

    • ==数据和程序逻辑独立（概念独立）==：
      

      当模式改变时，可用外模式/模式定义语句，调整外模式/模式映象定义，从而保持外模式不变
      

      举例：如图，修改属性（学号，姓名，性别）对应的变量名，数据逻辑的改编，并不会改变最后的结果

    2）模式/内模式映象
      

    • ==数据和程序物理独立（存储独立）==： 
      

      当存储结构改变时，可（由DBA）用内模式定义语句，调整内模式定义，从而保持模式不变
